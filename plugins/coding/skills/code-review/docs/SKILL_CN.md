# Code Review 代码审查

当用户请求 /code-review、想要自动化 PR 审查，或审查本地未提交变更时使用此 skill。

## 模式检测

首先确定审查模式：
- **PR 模式**：用户提供 PR 编号、URL，或明确要求审查 PR
- **本地模式**：用户要求审查本地变更、未提交变更、暂存变更，或未指定 PR

## 工作流程

### 步骤 1：收集变更

**PR 模式：**
1. 资格检查：
   - 如果 PR 已关闭、是草稿、是琐碎或自动化的、或已被你审查过，则跳过。
2. 使用 `gh pr diff` 获取 PR diff

**本地模式：**
1. 检查变更：
   - 使用 `git status` 查看变更内容
   - 使用 `git diff HEAD` 获取所有未提交变更（暂存 + 未暂存）
   - 如果没有变更，通知用户并退出
2. 收集受影响的文件路径用于 CLAUDE.md 查找

### 步骤 2：收集指导

返回相关 CLAUDE.md 文件的路径：根 CLAUDE.md 加上变更涉及目录中的任何 CLAUDE.md。

### 步骤 3：总结变更

### 步骤 4：独立审查（5 个并行代理）

使用 Agent 工具并发启动所有 5 个代理，设置 `run_in_background: true`。代理将并行运行，每个完成时会自动通知你。

**重要**：不要使用 TaskOutput 等待代理。让它们在后台运行，完成时会自动收到通知。

每个代理返回问题及理由：
- 代理 1：CLAUDE.md 合规性。
- 代理 2：仅 diff 中的明显 bug（避免额外上下文）。
- 代理 3：git blame 或历史上下文。
- 代理 4：涉及这些文件的先前 PR/提交（本地模式使用 `git log`）。
- 代理 5：修改文件中的代码注释。

### 步骤 5：置信度评分

使用此标准对每个问题评分 0-100：
- 0：完全不自信。误报或预先存在的问题。
- 25：有些自信。可能是真的，可能是误报。
- 50：中等自信。是真的但次要或不太可能。
- 75：高度自信。真的、重要的，或明确在 CLAUDE.md 中。
- 100：绝对确定。肯定是真的且频繁发生。

对于 CLAUDE.md 问题，验证指导是否明确指出该问题。

### 步骤 6：过滤和输出

过滤掉分数 < 80 的问题。然后根据模式输出结果：

**PR 模式：**
- 发布前重新检查资格
- 用 `gh` 发布简要的审查评论

**本地模式：**
- 将结果写入 `docs/code-review/{datetime}/review.md`，datetime 格式为 `YYYY-MM-DD_HH-mm-ss`
- 向用户输出摘要和文件路径

## 审查标准

评估代码时使用以下准则：

### 工程原则

- **DRY** —— 积极标记代码重复。
- **测试** —— 经过良好测试的代码是不可协商的；宁愿测试太多也不要测试太少。
- **适度工程化** —— 不是工程化不足（脆弱、hacky）也不是过度工程化（过早抽象、不必要的复杂性）。
- **边缘情况** —— 倾向于处理更多边缘情况，而不是更少；深思熟虑 > 速度。
- **明确优于聪明** —— 清晰胜过技巧。
- **最小差异** —— 用最少的新的抽象和触及的文件来实现目标。

### 文档和图表

- **ASCII 图表** —— 非常重视，用于数据流、状态机、依赖图、处理管道和决策树。在设计文档中自由使用。
- **复杂逻辑** —— 在代码注释中适当位置嵌入 ASCII 图表：Models（数据关系、状态转换）、Controllers（请求流）、Services（处理管道）、Tests（设置内容及其原因）。
- **图表维护** —— 修改附近有 ASCII 图表注释的代码时，验证图表是否仍然准确。作为同一提交的一部分更新它们。陈旧的图表比没有图表更糟 —— 它们会主动误导。在审查期间标记遇到的任何陈旧图表，即使它们在直接范围之外。

## 要避免的误报

- 预先存在的问题或变更之外的代码。
- 看起来像 bug 但实际上不是的东西。
- CLAUDE.md 中未指出的迂腐挑剔或风格问题。
- linter、类型检查器或 CI 会捕获的问题（导入、格式、类型错误）。
- 一般质量问题，除非 CLAUDE.md 明确要求。
- 代码中明确静音的问题（lint ignore 注释）。
- 与变更目标匹配的故意行为更改。

## 注意事项

- 不要运行构建、测试或类型检查。
- PR 模式使用 `gh` 进行 GitHub 交互。
- 先制作待办列表。
- 引用并链接每个问题。

## 输出格式

使用 `templates/` 目录中的模板：
- **PR 模式**: `templates/pr-comment.md`
- **本地模式**: `templates/local-review.md`

### 本地模式

将结果写入 `docs/code-review/{datetime}/review.md`，datetime 格式为 `YYYY-MM-DD_HH-mm-ss`。

**重要**: 输出内容使用中文。

写入后向用户输出：
```
代码审查完成。结果已保存至: docs/code-review/{datetime}/review.md
```

### PR 模式

使用 `gh` 发布评论。如果过滤后没有问题，除非被要求否则不评论。

## 链接/引用格式要求

**PR 模式：**
- 使用完整的 git SHA。
- 使用 `#Lstart-Lend` 并至少有一行上下文。
- 使用正在审查的 PR 的仓库名称。

**本地模式：**
- 使用带有行号的文件路径，格式为 `path:line` 或 `path:start-end`。
- 引用 diff 中的实际行号。
